# GraphDB 性能基准测试计划

**文档版本**: v1.0  
**更新日期**: 2026-06-18  
**目标**: 建立完整的性能基准测试框架，识别和解决项目中的性能瓶颈

---

## 1. 概述

### 1.1 目标

建立系统化的性能基准测试体系，用于：
- 发现关键性能瓶颈
- 监控性能变化趋势
- 指导架构优化和性能改进
- 确保版本发布的性能质量

### 1.2 范围

- **核心模块**: storage、transaction、query、search、api
- **关键操作**: 图数据增删改查、事务处理、全文搜索、向量搜索
- **性能维度**: 延迟、吞吐量、内存占用、磁盘IO

### 1.3 预期收益

| 预期收益 | 说明 |
|--------|------|
| **性能基线** | 建立各关键操作的性能基线，为后续优化提供对比基准 |
| **瓶颈识别** | 通过对比分析，精准定位性能瓶颈 |
| **性能追踪** | 持续监控性能变化，及时发现性能回退 |
| **优化指导** | 基于数据驱动的优化决策，提高投资回报率 |

## 2. 关键性能指标（KPI）

### 2.1 响应时间（Latency）

| 操作 | 目标 | 说明 |
|------|------|------|
| 顶点插入 | <1ms | 单个顶点插入 |
| 顶点查询 | <1ms | 按ID查询单个顶点 |
| 边插入 | <1ms | 单条边插入 |
| 边查询 | <2ms | 单条边或邻接表查询 |
| 简单图查询 | <10ms | Cypher/Gremlin 简单查询 |
| 全文搜索 | <100ms | 返回1000条结果 |
| 向量搜索 | <50ms | 返回10条最相似向量 |

### 2.2 吞吐量（Throughput）

| 操作 | 目标 | 说明 |
|------|------|------|
| 批量插入 | >100k ops/s | 批量顶点/边插入（事务内） |
| 批量查询 | >50k ops/s | 批量顶点查询 |
| 并发写 | >10k ops/s | 4线程并发写入 |
| 并发读 | >100k ops/s | 8线程并发读取 |

### 2.3 资源占用

| 指标 | 监控点 |
|------|--------|
| 内存 | 1M顶点的内存占用 |
| 磁盘 | 1M顶点的存储大小 |
| CPU | 各操作的CPU使用率 |

## 4. 性能瓶颈识别方法论

### 4.1 性能分析维度

#### 4.1.1 CPU 瓶颈识别

```
识别方法：
1. CPU占用率 >80% 且吞吐量没有线性增长
2. 通过perf/flamegraph分析热点函数
3. 对比不同大小数据集的CPU占用增长曲线

常见的CPU瓶颈：
- 频繁的内存分配/释放
- 不必要的复制操作
- 低效的数据结构查找
- 没有充分利用SIMD指令
```

#### 4.1.2 内存瓶颈识别

```
识别方法：
1. 内存占用增长率超过理论值
2. 内存碎片率过高（>20%）
3. 缓存命中率低（<50%）
4. GC/分配耗时过长

常见的内存瓶颈：
- 不必要的中间数据结构
- 缓存设置不合理
- 内存池使用不当
- 频繁的大对象分配
```

#### 4.1.3 IO 瓶颈识别

```
识别方法：
1. 磁盘IO等待时间 >10% 总耗时
2. 缓冲区命中率低
3. 随机IO比例过高
4. 大量的小IO请求

常见的IO瓶颈：
- 缺少缓冲/批处理
- 不合理的数据分片
- 缺少索引导致全表扫描
- 写操作没有异步处理
```

### 4.2 基准测试对比分析

#### 4.2.1 纵向对比（版本对比）

```
对比基准测试结果：
新版本 vs 基线版本

关键指标：
- 延迟变化：±5% 以内为可接受
- 吞吐量变化：±5% 以内为可接受
- 内存占用变化：±10% 以内为可接受

触发告警的条件：
- 延迟增加 >10%
- 吞吐量下降 >10%
- 内存占用增加 >20%
```

#### 4.2.2 横向对比（配置对比）

```
对比不同配置下的性能：
- 缓存大小：0MB vs 100MB vs 1GB
- 线程数：1 vs 4 vs 8 vs 16
- 批量大小：1 vs 10 vs 100 vs 1000
- 压缩算法：none vs zstd vs lz4

找出最优配置及其性能收益。
```

### 4.3 扩展性分析

#### 4.3.1 吞吐量扩展性

```
评估：当数据量/并发度增加时，吞吐量的变化

理想情况：
- 线性扩展：吞吐量随并发度线性增长
- 当吞吐量不再增长或下降时，说明遇到瓶颈

测试场景：
数据量: 1k -> 10k -> 100k -> 1M
并发度: 1 -> 2 -> 4 -> 8 -> 16

分析：
- 找出开始下降的临界点
- 分析该临界点的根本原因
```

#### 4.3.2 延迟扩展性

```
评估：当数据量/并发度增加时，延迟的变化

理想情况：
- 延迟保持稳定（O(1) 操作）
- 或按对数增长（O(log n) 操作）

危险信号：
- 延迟随数据量呈二次增长
- 延迟随并发度呈线性增长
- 延迟出现长尾现象

关键指标：
- P50, P95, P99 延迟趋势
- 延迟标准差和最大值
```

## 5. 性能瓶颈优化指南

### 5.1 常见瓶颈和解决方案

| 瓶颈类型 | 表现 | 根本原因 | 优化方案 |
|--------|------|--------|--------|
| **内存分配瓶颈** | CPU占用高但吞吐不增长 | 频繁的小对象分配 | 内存池、对象复用、预分配 |
| **缓存失效** | 内存占用大但缓存命中率低 | 缓存策略不当 | LRU、时间衰减、自适应大小 |
| **锁竞争** | 并发性能下降 | 共享数据上的频繁竞争 | 分片锁、无锁数据结构、局部化 |
| **IO阻塞** | 延迟波动大 | 同步IO、缺少缓冲 | 异步IO、预读、批处理 |
| **查询计划差** | 查询性能与预期不符 | 成本估算不准 | 动态规划、启发式搜索 |
| **数据结构低效** | 某操作延迟持续高 | 使用了不合适的数据结构 | 算法优化、SIMD、并行化 |

### 5.2 优化前的验证清单

在进行优化前，确保已完成：

- [ ] 建立基线基准测试
- [ ] 已通过profiling工具确认热点
- [ ] 已分析根本原因
- [ ] 已评估优化的收益/成本
- [ ] 已设计测试用例验证优化
- [ ] 已准备回滚方案

## 7. 性能基准测试最佳实践

### 7.1 基准测试编写规范

```rust
// ✓ 正确做法
#[bench]
fn bench_vertex_insert(b: &mut Bencher) {
    // 1. 初始化（每次测试）
    let storage = setup_storage();
    
    // 2. 预热（不计时）
    storage.insert_vertex(&Vertex::default());
    
    // 3. 实际基准测试（计时）
    b.iter(|| {
        black_box(storage.insert_vertex(&create_test_vertex()))
    });
    
    // 4. 清理
    cleanup_storage(storage);
}

// ✗ 错误做法
#[bench]
fn bench_vertex_insert_bad(b: &mut Bencher) {
    // 错误1: 初始化放在 iter 里
    b.iter(|| {
        let storage = setup_storage();
        storage.insert_vertex(&create_test_vertex());
        cleanup_storage(storage);
    });
}
```

### 7.2 数据生成最佳实践

```rust
// ✓ 预生成数据
let test_data = (0..10000)
    .map(|i| create_vertex(i))
    .collect::<Vec<_>>();

b.iter(|| {
    test_data.iter().for_each(|v| {
        black_box(storage.query_vertex(v.id()));
    })
});

// ✗ 避免在测试中生成数据
b.iter(|| {
    let v = create_vertex(rand::random());  // 随机性影响结果
    storage.query_vertex(v.id());
});
```

### 7.3 并发基准最佳实践

```rust
use std::sync::Arc;

#[bench]
fn bench_concurrent_read(b: &mut Bencher) {
    let storage = Arc::new(setup_storage());
    let handles = vec![];
    
    b.iter(|| {
        // 生成并发任务
        for _ in 0..8 {
            let storage = Arc::clone(&storage);
            let handle = std::thread::spawn(move || {
                for _ in 0..100 {
                    black_box(storage.query_vertex(1));
                }
            });
            handles.push(handle);
        }
        
        // 等待所有任务完成
        for handle in handles.drain(..) {
            handle.join().unwrap();
        }
    });
}
```

### 7.4 避免常见陷阱

| 陷阱 | 影响 | 避免方法 |
|------|------|--------|
| 编译器优化消除计算 | 结果不真实 | 使用 `black_box()` |
| GC/内存分配影响结果 | 结果波动大 | 预分配，避免分配 |
| 缓存预热不足 | 首次运行很慢 | 在 iter 前预热 |
| 测试间相互影响 | 结果不独立 | 隔离测试数据 |
| 过短的测试时间 | 结果不稳定 | 配置足够长的运行时间 |
| 忽视外部因素 | 结果不可重复 | 关闭后台程序、固定CPU频率 |
